# Требования

Документ фиксирует функциональные и нефункциональные требования.

## Функциональные требования (MVP)
1. Создание, изменение и удаление справочников.
2. Создание, изменение и удаление атрибутов.
3. Настройка набора атрибутов для конкретного справочника.
4. Валидация объектов по правилам атрибутов.
5. CRUD объектов справочника.
6. Публикация событий в Kafka для всех операций create/update/delete.
7. Аудит изменений (кто/когда/что менял).
8. Синхронизация данных в OpenSearch для поиска.

## Текущий статус реализации (backend, март 2026)
Реализовано:
1. CRUD справочников, атрибутов, схемы и объектов.
2. Валидация объектов по схеме (required/type/reference/enum/validators/unique).
3. RBAC по ролям из `X-User-Role`.
4. Аудит изменений в `audit_events` и чтение аудита через API.
5. Интеграционные тесты с тестовой PostgreSQL (`make test-integration`).
6. Полный e2e backend сценарий (автозапчасти) с покрытием типов данных, CRUD, поиска и edge-cases.

В работе/запланировано:
1. Outbox publisher в Kafka (фоновый воркер и доставка событий).
2. Синхронизация поискового слоя (OpenSearch).
3. Метрики и наблюдаемость для outbox/indexing.

## Результаты e2e (2026-03-17)
Подтверждено:
1. Корректная работа CRUD для dictionaries/attributes/schema/entries.
2. Корректная валидация типов `string`, `number`, `date`, `boolean`, `enum`, `reference`, а также multivalue-полей.
3. Работоспособность поиска (`eq`, `ne`, `contains`, `prefix`, `in`, `lt`, `lte`, `gt`, `gte`, `range`) и пагинации.

Обнаруженные аномалии:
1. Сортировка по числовому полю в поиске может работать лексикографически (строковое сравнение), а не как числовая.
2. При удалении справочника (`HTTP 204`) в серверном логе возможна ошибка записи audit-события по FK (`audit_events_dictionary_fk`).

## Нефункциональные требования
1. Масштабируемость: 100 млн+ объектов в справочнике.
2. Надежность: отсутствие потери событий между Postgres и Kafka.
3. Консистентность: Postgres — источник истины.
4. Время отклика: запись должна завершаться за секунды.
5. Наблюдаемость: метрики задержек outbox и индексации.
6. Безопасность: роли и права доступа для административных операций.

## Уточнения по аудиту и безопасности (MVP)
### Аудит
1. Для каждой успешной операции `create/update/delete` по справочникам, атрибутам, схеме и объектам создается запись в `audit_events`.
2. В аудите фиксируются: `request_id`, кто выполнил действие (`actor_external_id` из заголовка `X-User-Id`), что изменилось (`entity_type`, `entity_id`, `action`), время, состояние до/после (`before_state`, `after_state`) и служебные метаданные запроса.

### Роли и доступ (RBAC)
1. Базовые роли MVP:
- `mdm_admin` — полный доступ к административным операциям.
- `mdm_editor` — изменение справочников, атрибутов, схемы и объектов.
- `mdm_viewer` — только чтение данных.
- `mdm_auditor` — чтение журнала аудита.
2. MDM backend расположен за доверенным gateway, интегрированным с Keycloak.
3. Gateway после проверки токена передает в MDM заголовки:
- `X-User-Id` — идентификатор пользователя (пример: `X-User-Id: 100`).
- `X-User-Role` — роли пользователя через запятую (пример: `X-User-Role: mdm_admin,mdm_editor`).
4. Отсутствие обязательных заголовков аутентификации возвращает `401`, недостаточность прав роли — `403`.
5. В MDM нет локальных таблиц `users/roles`; backend не обращается к внешним сервисам за ролями.

### Идентификаторы (ID)
1. В бизнес-таблицах используются UUID.
2. Первичные ключи `audit_events.id` и `outbox_events.id` — UUID.

## Целевые характеристики
- Доступность: 99.5%+ (внутренний сервис).
- RPO: близкий к нулю (через outbox).
- RTO: часы (допустимо для внутренней системы).

## После MVP

3. Интеграции с сервисами нормализации.
4. Расширенный поиск и ранжирование.
