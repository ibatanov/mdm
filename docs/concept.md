# Концепция MDM/PIM

Этот документ фиксирует начальную концепцию и архитектуру универсальной MDM/PIM системы.

## Цель
Сделать гибкую систему, где:
- справочники создаются динамически;
- атрибуты имеют типы данных и правила валидации;
- каждый справочник определяет, какие атрибуты в нем допустимы;
- все операции create/update/delete публикуются в Kafka;
- система может масштабироваться до 100 млн+ объектов в справочнике.

## Объем MVP
Входит:
- CRUD справочников;
- CRUD атрибутов (тип, обязательность, значение по умолчанию, валидаторы, ссылки);
- настройка схемы справочника (разрешенные атрибуты);
- CRUD объектов справочника с валидацией;
- outbox и публикация событий в Kafka;
- синхронизация в OpenSearch для поиска.

Не входит (MVP):
- поиск дубликатов;
- массовые изменения;
- интеграции с сервисами нормализации.

## Архитектура (монолит, модульный)
Логические модули:
- Meta: справочники, атрибуты, схема;
- Entries: CRUD, валидация;
- Outbox: надежная доставка событий в Kafka;
- Search Sync: индексация в OpenSearch;
- Admin API: REST эндпоинты для операторов.

### Поток данных
1. Запись в Postgres в рамках транзакции.
2. Запись события в outbox в той же транзакции.
3. Фоновый воркер публикует события в Kafka.
4. Консьюмеры обновляют индексы OpenSearch.

### Стратегия хранения
- Postgres — источник истины.
- OpenSearch — поисковый слой для чтения.
- JSONB используется для гибкости и быстрых итераций.

## Масштабирование
- Партиционирование таблицы объектов (по dictionary_id или по хэшу).
- Кэширование метаданных атрибутов в памяти.
- Массовая индексация в OpenSearch.
- Минимум кросс-справочных join в горячих сценариях.

## После MVP
- Поиск дубликатов (вероятностный + правила).
- Массовые операции (асинхронные джобы).
- Нормализация через внешние сервисы.
