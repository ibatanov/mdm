# UI (Vue) — требования для MVP

Актуальность: март 2026.

## 1. Цель
Сделать отдельный подпроект `ui` (веб-интерфейс) для работы с MDM API:
- покрыть все текущие публичные ручки backend;
- обеспечить удобную работу операторов с ролями `mdm_admin`, `mdm_editor`, `mdm_viewer`, `mdm_auditor`;
- зафиксировать современный, легкий, светлый визуальный стиль.

## 2. Технологический стек (обязательно)
- `Vue 3` (последняя стабильная версия на момент старта реализации).
- `TypeScript`.
- `Vite`.
- `Vue Router`.
- `Pinia` для клиентского состояния.
- HTTP-клиент (`fetch`/`axios`) с единым API-слоем.

Рекомендуется:
- `@tanstack/vue-query` для server-state и кэширования запросов.
- `Vitest` + `Vue Test Utils` для unit/integration.
- `Playwright` для e2e smoke.

## 3. Интеграционный контракт с backend
- Базовый API путь: `/api/v1`.
- Служебные ручки: `/healthz`, `/readyz`.
- Backend находится за gateway (Keycloak).
- UI не управляет локальными ролями/пользователями в MDM.
- Gateway должен передавать в backend:
  - `X-User-Id`
  - `X-User-Role`
- `X-Request-Id` опционален: backend генерирует при отсутствии/невалидном значении.

### 3.1 Dev-режим до подключения gateway/Keycloak
Пока gateway и Keycloak не подключены, UI должен уметь эмулировать пользователя для разработки.

Обязательные правила:
- В `development` UI добавляет в запросы заголовки:
  - `X-User-Id`
  - `X-User-Role`
- Источник значений:
  - либо `.env` переменные (`VITE_DEV_USER_ID`, `VITE_DEV_USER_ROLES`);
  - либо dev-переключатель профиля в UI (только для локальной разработки).
- Минимальные профили для быстрого тестирования:
  - `admin`: `X-User-Id=100`, `X-User-Role=mdm_admin`
  - `editor`: `X-User-Id=101`, `X-User-Role=mdm_editor`
  - `viewer`: `X-User-Id=102`, `X-User-Role=mdm_viewer`
  - `auditor`: `X-User-Id=103`, `X-User-Role=mdm_auditor`
- В `production` эмуляция должна быть отключена.
- Dev-эмуляция не должна подменять/перекрывать заголовки, если они уже выставлены gateway.

## 4. Функциональный scope UI (MVP)
UI должен поддерживать все ручки, которые уже реализованы в backend.

### 4.1 Справочники
- `POST /dictionaries`
- `GET /dictionaries`
- `GET /dictionaries/{dictionary_id}`
- `PATCH /dictionaries/{dictionary_id}`
- `DELETE /dictionaries/{dictionary_id}`

### 4.2 Атрибуты
- `POST /attributes`
- `GET /attributes`
- `GET /attributes/{attribute_id}`
- `PATCH /attributes/{attribute_id}`
- `DELETE /attributes/{attribute_id}`

### 4.3 Схема справочника
- `GET /dictionaries/{dictionary_id}/schema`
- `PUT /dictionaries/{dictionary_id}/schema`

### 4.4 Объекты (entries)
- `POST /dictionaries/{dictionary_id}/entries`
- `GET /dictionaries/{dictionary_id}/entries`
- `GET /dictionaries/{dictionary_id}/entries/{entry_id}`
- `PATCH /dictionaries/{dictionary_id}/entries/{entry_id}`
- `DELETE /dictionaries/{dictionary_id}/entries/{entry_id}`
- `POST /dictionaries/{dictionary_id}/entries/search`

### 4.5 Аудит
- `GET /audit/events`

### 4.6 Здоровье сервиса
- `GET /healthz`
- `GET /readyz`

## 5. Роли и поведение UI
- `mdm_admin`: полный доступ ко всем операциям.
- `mdm_editor`: CRUD справочников, атрибутов, схемы, объектов.
- `mdm_viewer`: только чтение справочников/атрибутов/схемы/объектов.
- `mdm_auditor`: доступ к аудиту.

Требования к интерфейсу:
- скрывать или дизейблить недоступные действия по роли;
- показывать понятные сообщения при `403 forbidden`;
- не дублировать серверные проверки прав на клиенте как «источник истины».

## 6. Информационная архитектура UI
- `Справочники`:
  - список и создание справочников;
  - карточка справочника (`name/description`);
  - настройка атрибутов справочника (схема: required/unique/multivalue/validators/position).
- `Атрибуты`:
  - каталог атрибутов;
  - создание/редактирование/удаление;
  - отображение, в каких справочниках атрибут участвует.
- `Объекты`:
  - CRUD объектов выбранного справочника по динамической форме (из схемы);
  - поиск по выбранному справочнику с фильтрами и сортировкой;
  - отдельный глобальный поиск по всем объектам всех справочников.
- `Аудит` (лента событий аудита с фильтрами).
- `Health` (`/healthz`, `/readyz`).

## 7. UX требования
- Единый layout: верхняя панель + боковая навигация + рабочая область.
- Любая мутация (`create/update/delete`) завершается:
  - уведомлением об успехе/ошибке;
  - актуализацией списков и карточек.
- Все списки: пагинация (`limit`, `offset`) + отображение `total`.
- Формы: inline-валидация полей + вывод серверных ошибок `422`.
- Для `entries` поддержать динамическую форму по схеме справочника:
  - типы `string/number/date/boolean/enum/reference`;
  - `required/is_unique/is_multivalue`;
  - валидаторы из `validators`.
- Для `PATCH entries`: при явном `null` удалять поле (с подтверждением в UI, где уместно).

## 8. Визуальный стиль (дизайн-направление)
- Общая стилистика: современный светлый интерфейс (`light first`).
- Референс-паттерн: `Berry/Figma Admin Dashboard` (table-first рабочий экран, компактный sidebar, мягкие серые поверхности, синий primary).
- Цвета:
  - светлая база и нейтральные поверхности (`gray/cool-gray` скейлы);
  - акцентный синий для primary-action (кнопки, активные состояния, ссылки);
  - отдельные семантические цвета для success/warning/error/info.
- Типографика:
  - качественный sans-serif шрифт для UI (плотный, dashboard-ориентированный);
  - отдельный моноширинный для UUID/технических значений;
  - иерархия заголовков, плотная читаемая сетка.
- Компоненты:
  - мягкие тени, аккуратные радиусы, четкие состояния hover/focus/active;
  - clean data-table с верхним тулбаром фильтров/поиска;
  - карточки форм в стиле admin console (светлые контейнеры, явные action-кнопки);
  - модальные/оверлейные действия для сложных операций редактирования.

Рекомендуемые шрифты:
- Основной: `Manrope` или `IBM Plex Sans`.
- Моно: `JetBrains Mono`.

## 9. Анимации и микровзаимодействия
- Легкие и быстрые анимации (`120–220ms`, без «пружинной» перегрузки).
- Разрешены:
  - fade/slide для диалогов и панелей;
  - мягкие hover/press эффекты кнопок;
  - skeleton/loading state для таблиц и карточек.
- Запрещены:
  - тяжелые/длительные анимации, ухудшающие рабочий процесс.

## 10. Компонентная модель (базовый набор)
- `AppShell`, `Sidebar`, `TopBar`.
- `DataTable`, `Pagination`, `FilterBar`.
- `FormField` (text, number, date, checkbox, select, multivalue).
- `Drawer/Modal`, `ConfirmDialog`.
- `Toast/Alert`, `EmptyState`, `ErrorState`, `Skeleton`.
- `JsonPreview` для `before_state/after_state` в аудите.

## 11. Доступность и качество
- Минимум `WCAG 2.1 AA` по контрастам и фокусу.
- Полная клавиатурная навигация для форм и таблиц.
- Семантическая разметка и aria-атрибуты для основных интерактивных блоков.
- Локализация MVP: русский язык интерфейса.

## 12. Нефункциональные требования к UI
- Адаптивность: desktop first, корректная работа от `1280px` и выше, приемлемо на `>=768px`.
- Производительность:
  - lazy-load роутов;
  - отсутствие блокирующих длинных операций в main-thread;
  - кэширование списков и карточек.
- Наблюдаемость (минимум):
  - единый лог ошибок API на клиенте;
  - отображение `request_id` в ошибках для упрощения диагностики.

## 13. Критерии готовности MVP UI
1. Доступны все разделы и все текущие backend-ручки.
2. Роли корректно ограничивают действия в интерфейсе.
3. Формы и таблицы стабильно работают с `400/401/403/404/409/422/500`.
4. Дизайн соответствует направлению: светлая тема, современная типографика, легкие анимации, аккуратные тени.
5. Есть базовые unit/e2e smoke тесты критичных сценариев.
6. В dev-режиме поддерживается переключение/эмуляция ролей без gateway; в production эта возможность отключена.
